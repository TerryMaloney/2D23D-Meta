# Real-World Validation Report

**Date:** June 11, 2026 (updated after parser hardening — see history at bottom)
**Scope:** legitimately sourced public statement documents + the privacy-safe
mechanism for validating private statements locally.

## Honest framing, first

No real customer statement has been parsed in this repository, by design —
private statements never enter the codebase or this development environment.
The corpus below consists of **bank-published specimens, educational teaching
samples, and a government model form**. These are real-world *layouts* (two
were generated by actual bank statement systems) but they are not real
accounts. Accuracy claims are computed only from this labeled corpus.

Every entry is exercised by `packages/parser/test/real-corpus.test.ts`
(cache-only files are skipped gracefully when absent, e.g. in CI).

## Private statement validation (the mechanism owners actually use)

Real coverage comes from the owner running, locally:

```bash
npm run validate:statements
```

against PDFs placed in `.private-statement-validation/input/` (gitignored).
The validator parses with the production engine and reports **only**:
anonymous IDs, page counts, detected institution/template, type, transaction
counts, reconciliation status, difference in cents, flagged-row counts, error
categories, runtimes, and aggregates — never descriptions, names, account
numbers, balances, amounts, or any extracted text. Output stays in the
gitignored folder; nothing is transmitted anywhere (the script performs no
network I/O).

Verified by test (`packages/parser/test/anonymize.test.ts`) and by grep: a
validator run over fixtures containing known merchant strings and amounts
produced output with zero occurrences of any of them.

## Layout twins (turning private failures into public fixtures)

A failed private statement can be converted into a **layout twin**: the
anonymizer (geometry kept; words hashed; dates replaced with valid dates of
the same format; account-number digit runs and SSN shapes stripped; years
kept) feeds `packages/parser/fixtures/layout-twin.ts`, which regenerates a
synthetic PDF with the same column positions and section structure but
**internally consistent fake values** — opening + transactions = closing,
chained running balances, matching printed totals. Twins parse through the
production engine and reconcile `verified`, so they work as committable
regression fixtures.

Sanitization guarantees are tested: seeded PII patterns (names, addresses,
SSN shapes, 10-digit account numbers, exact amounts) do not survive into the
anonymized layout or the twin PDF's extracted text.

## Limitations

- The public corpus is small; legitimate, redistributable statement specimens
  are genuinely scarce. The committed subset (3 documents) is what CI can
  exercise; the rest is reproducible via manifest URLs.
- Educational graphics and model forms are adversarial inputs, not target
  inputs — their value is proving the engine fails *closed* (typed errors,
  no invented transactions) rather than parsing them.
- No scanned-statement support: SaskMoney's image-only PDF correctly triggers
  `SCANNED_PDF` rather than being misread.
